# Системная диагностика фронтенд-багов

От фенотипа к корню.

## Когда активировать

- Пользователь сообщает о баге в UI
- Баг воспроизводится визуально, но не очевидна причина
- После первой неудачной попытки фикса (перекос в одну сторону)

## Методика: от фенотипа к корню

Не начинай с гипотезы о слое (данные, логика, рендер, стили).
Начни с **фенотипа бага** — что именно видит пользователь.

### Диагностическая пирамида (снизу вверх)

```
┌─────────────────────────┐
│ 7. Server/API           │ ─── данные с сервера
│ 6. State management     │ ─── стейт, хуки, контекст
│ 5. Data filtering/logic │ ─── фильтры, сортировка, useMemo
│ 4. Component render     │ ─── ререндер, memo, keys
│ 3. DOM state            │ ─── атрибуты, классы, data-*
│ 2. CSS / styling        │ ─── opacity, display, visibility, z-index
│ 1. Фенотип бага         │ ─── что видит пользователь
└─────────────────────────┘
```

### Порядок диагностики

1. **Собери фенотип** — что именно не так? "Пустота"? "То же самое"? "Мигает"?
2. **Данные** — console.log в нужном месте. Данные приходят/возвращаются?
3. **DOM** — открой Elements в DevTools. Элемент есть в DOM? Какие у него классы/стили?
4. **CSS** — какой computed style у элемента? opacity, display, visibility?
5. **Рендер** — компонент ререндерится? React DevTools, console.log в теле компонента.
6. **Логика** — что происходит с состоянием при действии? Следи за изменениями.
7. **API** — правильный запрос? Правильный ответ?

### Для каждого шага задавай вопрос

> "Я проверил слой X? А что если проблема в слое Y?"

Не застревай в одном слое — если данные корректны, смотри DOM/CSS.

## Анти-паттерны

### ❌ Перекос в логику данных

*Симптом:* услышав "контент не возвращается", сразу лезешь в фильтры,
useMemo, debounce, не проверив, отображается ли элемент вообще.

*Последствия:* теряешь время, пользователь раздражается.

*Исправление:* console.log результата → если данные верны, иди в DOM/CSS.

### ❌ Игнорирование CSS-классов и атрибутов

*Симптом:* не замечаешь `data-reveal`, `opacity: 0`, IntersectionObserver.

*Последствия:* лечишь логику, а проблема в том, что элемент невидим.

*Исправление:* открывай Elements → проверяй атрибуты и стили элемента.

### ❌ Перерендер vs классы

*Симптом:* думаешь, что React сам восстановит DOM-состояние.

*Реальность:* `opacity`, анимации, классы от IntersectionObserver — не
восстанавливаются автоматически при ререндере.

## Кейс: "Поиск вдохновения не возвращает карточки"

*Фенотип:* пользователь вводит поиск, шаблоны фильтруются. Нажимает
крестик — пустой чёрный фон, хотя data возвращаются (console.log показал
resultCount: 4).

*Что пошло не так:* я застрял на слое 5 (фильтрация в useMemo) и не
проверил слои 3-4 (DOM/CSS). Элементы были в DOM, но имели класс `reveal`
с `opacity: 0` и не имели `reveal--visible`.

*Корень:* IntersectionObserver добавляет `reveal--visible` только один раз
при монтировании. При ререндере (очистка поиска) React пересоздаёт DOM,
класс теряется, а Observer не перезапускается (зависимость `[activeCategory]`
не включает `searchQuery`).

*Фикс:* добавить `searchQuery` в зависимости Observer.

*Правильный путь:* фенотип "пустота" → проверить DOM (есть, opacity: 0) →
CSS (reveal) → кто управляет классом (Observer) → почему не сработал
(зависимости useEffect).
